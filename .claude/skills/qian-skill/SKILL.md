# 钱学森系统科学方法论：用于复杂软件工程的 AI 协同脚手架

把钱学森（Qian Xuesen）的系统科学思想压缩成 AI 辅助编码的执行流程。**这不是哲学讲座，是一组诊断 + 工作流**：在动手写代码之前（以及在卡住时）做总体设计，避免还原论陷阱、局部优化破坏全局、忽视耦合带来的涌现行为。

适用范围既包括纯软件场景（大型代码库、多服务系统、跨模块重构、性能/稳定性问题），也包括软硬协同 / 科学仪器 AI 这类天然的耦合复杂系统。下文以软件工程为主轴，仪器场景作为延伸。

## 何时启用

只要任务有以下任一信号，主动启用：

- 改动会跨越多个文件、多个模块或多个服务
- 在大型代码库里加新特性或重构现有特性
- 性能 / 稳定性 / 正确性问题"本地看正常但整体异常"
- 架构选型有"全局 vs 局部"权衡（性能、可维护性、团队边界）
- 跨服务 / 跨进程 / 跨机器的调试，常规手段无效
- 给硬件 / 科学仪器开发 AI 控制 / 数据 / 校准层
- 用户描述里出现："多个子系统"、"耦合"、"实时"、"难以预测"、"端到端"、"反馈回路"、"系统级"、"做了 X 还是出 Y"

**不适用**（第一步诊断会让你立即退出本 Skill）：

- 单文件、单函数的实现或修改
- 隔离的算法题（HumanEval / LeetCode 风格）
- 单行 bug 修复、纯字符串处理、纯类型修正
- 自包含的小工具脚本
- 简单的配置改动、文档改动
- 上下文里已经有清晰的最小修复范围且不涉及跨模块

> 核心原则：本 Skill 是**诊断驱动**的——先判定系统类型，再决定方法论。错配代价比省略本 Skill 还高。**简单任务上滥用本 Skill = 浪费用户时间。**

## 工作流：五步骤

### 第一步：系统类型诊断（不要跳过）

钱学森把系统分四类。先做五题速诊，再决定后面是否继续：

| 题 | 答案选项 |
|---|---|
| Q1. 子系统/模块数量 | <10 / 10–100 / >100 |
| Q2. 是否分层级 | 无 / 弱 / 强（多层） |
| Q3. 子系统种类 | 同质 / 少数几种 / 多种异质 |
| Q4. 与环境/人的耦合 | 封闭 / 弱耦合 / 强开放（人在回路 / 物理环境耦合 / 外部 API 链路） |
| Q5. 行为可预测性 | 可解析推导 / 可仿真 / 仅可在线探索 |

**额外维度：系统层次**——同时判断你当前的工作落在**微观 / 介观 / 宏观**哪一层：

- **微观**：单一函数、单一文件、单一进程内部
- **介观**：模块、服务、子系统、单一团队
- **宏观**：端到端用户体验、SLO、业务 KPI、组织/生态

不同层次用不同工具。**最常见错误是层次错配**——在微观找宏观问题的根因（如在 diff 里找用户增长停滞的原因），或用宏观语言谈微观细节。详见 `references/systems-engineering-discipline.md` 第 4 节。

**分类对照表**：

| 类型 | 典型答案 | 适用方法 |
|---|---|---|
| **简单系统** | 模块少、同质、封闭、可解析 | 还原论：模块化 + 单元测试 + 类型 |
| **简单巨系统** | 模块多但同质、弱耦合 | 还原论 + 统计/分布式工程 |
| **复杂系统** | 多种异质、强耦合、封闭 | 还原 + 仿真 + 全局指标 |
| **开放复杂巨系统（OCGS）** | 多种异质、分层、强开放、行为难预测 | **综合集成法（必须）** |

**典型识别**：

- 单函数、CRUD endpoint 加字段、单文件 bug 修复 → 简单系统 → **退出本 Skill**
- 大规模微服务集群（同质服务、自动伸缩）→ 简单巨系统
- 编译器、数据库引擎、操作系统内核、ML 训练框架 → 复杂系统
- 大型软件产品（前端 + 多个后端服务 + ML 模型 + 数据管线 + 真实用户）→ OCGS
- 科学仪器 AI 控制（硬件 + 固件 + 控制环 + 数据采集 + 标定 + 物理环境 + 操作员）→ OCGS

**判定结果决定行为**：

- 判为**简单系统**：直接退出本 Skill，按常规编码即可。**不要**继续后面四步。
- 判为**简单巨系统**：用第三步的总体设计（轻量版），跳过第四步详细综合集成。
- 判为**复杂系统 / OCGS**：完整走完第二到第五步。

公开你的判定结果给用户看（一行话），让用户能立即纠正。

### 第二步：三层次结构定位

任何复杂工程问题同时存在三个层次。AI 编码最常见的失败模式是**只在底层写代码**，导致"做对了但方向错了"。

```
[ 哲学层 ]   目标的意义、价值判断、约束（业务、合规、用户体验）
    ↓
[ 基础理论 ]  数学/物理/算法本质、不变式、误差/失败模型、关键假设
    ↓
[ 应用技术 ]  具体实现：API、数据结构、算法、性能、依赖、部署
```

**做法**：动手前，把当前任务的三层各写一段话。如果**基础理论层是空的**（你写不出系统的不变式、关键假设、失败模式），那么应用层每多写一行，技术债就深一层。**先回到上一层**与用户对齐，再继续。

**软件工程典型坑**：
- 直接改业务代码（应用层）而不写清楚业务不变式（基础层）→ 改 A 触发 B 退化
- 优化性能（应用层）不分析瓶颈模型（基础层）→ 优化错地方
- 加一个新功能没问"为什么加 / 不加什么"（哲学层）→ 功能蔓延

**仪器/AI 典型坑**：直接训 end-to-end 网络（应用层）不分析仪器物理（基础层）→ OOD 一塌糊涂。

### 第三步：总体设计（钱学森"总体设计部"思想）

复杂系统 / OCGS **必须先做总体设计，再做子系统**。开工前产出的清单（写到 `docs/system-design.md`、PR 描述顶部，或文件顶部注释，后续 PR 持续回看）：

**总体设计清单**：

1. **系统边界**：列出系统内 / 环境 / 人（人是否在系统内？外部 API 是否可控？）
2. **子系统清单**：每个子系统/模块一句话讲清职责（避免名词堆砌）
3. **耦合矩阵**：建一个 N×N 表，标注子系统两两之间的耦合类型与强度。耦合类型至少包括：
   - **数据耦合**（共享 schema / 数据流 / 数据库表）
   - **控制耦合**（A 调用 / 触发 / 命令 B）
   - **时序耦合**（A 必须在 B 之前 / 之后 / 同步 / 限速）
   - **状态耦合**（共享可变状态、缓存、配置、feature flag）
   - **物理耦合**（共享硬件资源、热、电、机械）—— 仪器系统专属
   - **标定耦合**（A 的校准影响 B 的解释）—— 仪器系统专属
   - **反馈耦合**（A 的输出经过某条路径回到 A 的输入）
   - **人/社会耦合**（操作员、开发者、用户行为是耦合的一部分）
4. **关键接口契约**：耦合最强的几对，把接口写死：schema、时序、错误语义、降级行为、幂等性
5. **总体（系统级）性能指标（≤5 条）**：注意是系统级，不是子系统级。例：端到端延迟、SLO、转化率、识别召回率、用户操作步数
6. **不变式（3–5 条）**：系统必须永远成立的命题。例：因果性、能量/数据守恒、用户隔离、单调性、幂等性、原子性
7. **目标函数（target function）**：把"为什么做这个系统"压缩成 ≤3 个有优先级的、可度量的目标，并显式标注它们之间的取舍。没有目标函数 → 漫无目的地优化，每一项都"改善了一点"但整体没变好。注意 Goodhart 法则：一旦指标变成目标，它就不再是好指标。
8. **控制论三性**——任何运行中的系统都必须满足：
   - **可观性（observability）**：在任意时刻能否知道系统在做什么？（logs / metrics / tracing / event sourcing）
   - **可控性（controllability）**：能否在不重启的情况下改变行为？（feature flag / kill switch / hot-reload / graceful drain）
   - **稳定性（stability）**：扰动后能否收敛？（idempotency / retry semantics / queue bound / MTTR）

   这三性是源自控制论的硬约束，比"接口好不好看"重要得多。详见 `references/old-new-three-theories.md` 第 2 节、`references/systems-engineering-discipline.md` 第 1 节。

**反模式**：
- 直接画"分层架构图"就开干 → 没有耦合矩阵，必然出现跨层依赖
- 子系统各自最优 → 系统级失败（每个微服务 P99 50ms，链路 P99 2s 的典型例子）
- 没有系统级指标 → 优化时不知道为什么优化、是否值得

详见 `references/coupling-matrix.md`。

### 第四步：综合集成法（从定性到定量） + 大成智慧

复杂系统 / OCGS 不能纯定量解决，也不能纯定性拍脑袋。钱学森晚年把综合集成升华为**大成智慧**（Da Cheng Zhi Hui）：人—机—网协同 = 智慧。

**研讨厅体系的三体系一统合**（直接对应 AI 编码的三方协同）：

| 钱学森原型 | AI 编码对应 | 角色 |
|---|---|---|
| 专家体系（人） | 工程师 + 领域专家 + reviewer | 提出问题、判断价值、最终拍板 |
| 机器体系 | Claude / Copilot / LSP / profiler / CI / 形式化验证 | 加速搜索、生成、验证 |
| 知识体系 | 代码库 + docs + commit history + 内外文献 | 提供事实和先验 |
| **以人为主**（一统合） | Pair-coding / PR review / arch meeting | 三者循环，**人保留价值判断与最终决定** |

```
定性假设（领域直觉、业务知识、物理先验、用户故事）
       +
定量证据（profile、trace、benchmark、A/B 实验、回放数据、单元/集成测试、仿真）
       +
循环（人 ↔ AI ↔ 数据，逐步把"猜"变成"证"）
       ↓
带置信度的工程决策
```

**思维模式切换**——人在循环中实际使用三种思维（钱学森"思维科学"）：

- **抽象思维**：演绎、证明、类型推导（AI 最擅长）
- **形象思维**：画图、空间结构、整体模式（架构、依赖图、状态机）
- **灵感思维**：跨越定式的顿悟（卡住时必须切换，AI 最弱）

何时切换：grep 改 30 分钟没进展、AI 重复生成相似输出、"再看一遍就能找到"的感觉持续 ≥20 分钟 = 切换形象或诱导灵感。详见 `references/three-thinking-modes.md`。

**AI 编码版本的纪律**：

- 不要让 AI **脱离人类领域直觉单跑**（容易把"语法对的代码"当成"对的解"）
- 不要**只听人类直觉**而不验证（人对复杂系统的直觉常常错）
- 不要**只看数据**而忽略业务/物理先验（噪声会拟合出假相关）
- **关键反模式**：AI 在没有人类输入的情况下连续多步推理 + 自动执行高代价操作（部署 / 删数据 / 改 prod）—— 违背"以人为主"

**常用场景模板**：

- **跨子系统 bug**：先让用户/自己说出 ≥3 个定性假设 → 每个假设设计一个最便宜的诊断 → 按代价×排除力排序执行 → 收敛
- **加新特性**：先列业务不变式 + 已有接口契约 → 找最小侵入点 → 用测试或类型先固化新不变式 → 再实现
- **性能优化**：先定性瓶颈猜测 → profile 验证 → 改 → 量化复测 → 不达预期回到定性
- **架构选型**：列出 ≤3 个备选 → 对每个写出"它在哪些总体指标上更优 / 更差" → 选
- **ML 模块设计**：先列业务/物理先验（定性）→ 小数据训练 + 仿真增广（定量）→ 残差按先验分组分析 → 改进先验或架构

详见 `references/meta-synthesis-workflow.md`。

### 第五步：涌现与反馈检查

复杂系统 / OCGS **必然**有涌现行为：子系统层面正常、系统层面异常。AI 编码时常常看不到，因为视野停在 diff。

**涌现监测清单**（每次写完较大改动后过一遍）：

- 这个改动可能产生的**系统级副作用**是什么？（至少写出一个）
- 是否引入新的**反馈回路**？（A 影响 B，B 又影响 A，或经过外部回到 A）
- **时序假设**是否被打破？（之前同步现在异步？之前每秒一次现在每分钟一次？）
- **失败模式**是否改变？（之前快速失败现在静默重试？）
- **人/操作员/开发者的行为**会不会被这个改动改变？（UI 改动、开发者工具改动尤其要警惕）
- 改动是否**进入了下次训练 / 标定 / 优化的输入**？（隐式反馈回路）
- 是否接近**突变点**？（缓存命中率从 99% 跌到 95% 可能瞬间引发雪崩——参数渐变，状态跳变）

**精确词汇**（来自老三论 + 新三论）：
- 涌现来自**协同学**——找到**序参量**（≤5 个支配系统宏观状态的变量）
- 反馈来自**控制论**——区分负反馈（稳定）vs 正反馈（放大 / 雪崩）
- 稳定流来自**耗散结构**——系统不是静止地正确，是在持续输入下动态地正确
- 跳变来自**突变论**——找你系统的"临界点"和"滞后曲线"

详见 `references/old-new-three-theories.md`。

**如果发现涌现**：不要在原地"修一处"，要回到**接口契约 / 总体设计 / 不变式**层面改。否则同一个 bug 会换形式回来。

### 横切纪律：双想法 + 一次成功

不属于某一步，而是贯穿所有步骤的工程纪律（来自钱学森"两弹一星"经验）：

- **双想法**：每次设计都要**正想**（从目标到实现）+ **反想**（从失败到原因）。反向清单：什么输入下会爆？什么环境下会爆？什么并发模式下会爆？失败时是快速失败还是静默退化？谁会被通知？
- **一次成功心态**：当部署/操作代价高时（DB migration、改 prod 数据、跨服务协议、用户面 API 变更），切换到"上去就不能 in-place 修"的纪律——同行评审 ≥2 人、完整 walkthrough、失败模式分析、staging 完整复现。**反模式**是把高代价场景当低代价场景"快速迭代"。

详见 `references/systems-engineering-discipline.md` 第 2、3 节。

## 与 AI 编码工具的具体接口

当用户给你一个任务，按这个顺序工作：

1. **公开第一步诊断**——把判定一句话说清楚（"这是一个 X 类系统"）
2. 判为**简单系统**：明确说"本 Skill 不适用，直接用常规编码"，立即退出
3. 判为**复杂系统 / OCGS**：先写或读总体设计（第三步），让用户确认接口与不变式
4. 写代码时**显式标注三层次**：你现在在哪一层、为什么这一层是该动的层
5. 调试 / 排查时**走综合集成循环**（第四步），不要凭空猜
6. PR / 提交前过**涌现清单**（第五步）
7. 总体设计 / 耦合矩阵 / 不变式与代码**保持同步**——写到 `docs/system-design.md`、PR 描述、或文件顶部注释

## 例子

### 例 1：用户说"在我们这个 monorepo 加一个'订单退款'特性"

- **诊断**：复杂系统（订单 + 支付 + 库存 + 通知 + 财务对账 + 用户）
- **三层次**：
  - 哲学：退款的产品语义？谁能退？退到哪？是否需要审核？
  - 基础：退款的不变式（钱必须守恒、防双花、状态机的合法迁移）；与支付网关的契约
  - 应用：API、数据库 schema、消息事件
- **总体设计**：识别耦合最强的几对（订单↔支付：状态耦合 + 时序耦合；支付↔财务：数据耦合 + 时序耦合）；写出退款状态机的不变式
- **综合集成**：先和产品/财务/支付确认业务不变式（定性）→ 写状态机测试（定量）→ 实现 → 跑混沌测试验证状态机
- **涌现警报**：退款会进入对账系统 → 是否引入新的失败模式？是否影响库存？通知幂等吗？

### 例 2：用户说"我们的下单链路 P99 在高峰期飙到 5 秒，但每个微服务 P99 都正常"

- **诊断**：经典 OCGS / 复杂系统问题（局部正常 + 全局异常）
- **三层次**：
  - 哲学：是延迟问题还是吞吐问题？用户感受的是哪个？SLO 是什么？
  - 基础：链路延迟的可加性（关键路径）、串/并行结构、排队论
  - 应用：具体的微服务、网关、缓存、数据库
- **总体设计**：耦合矩阵识别串行依赖链；系统级指标 = 端到端 P99（不是单服务 P99）
- **综合集成**：先列 ≥3 个假设（H1 关键路径上某服务有"低概率高延迟"长尾、H2 缓存击穿、H3 数据库连接池在高峰耗尽、H4 跨服务重试放大）→ 链路 trace 验证 → 收敛
- **涌现警报**：每加一层缓存可能掩盖根因 / 失败时引发雪崩；扩容可能改变线程池压力

### 例 3：用户说"重构这个 monorepo 的构建系统"

- **诊断**：复杂系统（多包 + CI + 开发者 + 缓存基础设施）
- **三层次**：
  - 哲学：为什么重构？开发体验？CI 成本？发布安全？哪些**不**重构（用户面 API、版本号语义）
  - 基础：依赖图、缓存命中模型、增量构建语义
  - 应用：bazel / turbo / nx / 自研 选型
- **总体设计**：耦合矩阵 = 包依赖图 + CI 步骤拓扑；不变式 = 任何子图独立可构建
- **综合集成**：先 profile 现状（定量）→ 工程师定性归类（IO / 重复编译 / 测试串行）→ 改一类量化复测
- **涌现警报**：缓存策略改变会改变开发者本地行为（人在系统内）

### 例 4：用户说"帮我写一个把字符串列表去重并排序的函数"

- **诊断**：简单系统（单函数、纯逻辑、无外部耦合）
- **决定**：**本 Skill 不适用，立即退出**
- **直接给代码**：3-5 行普通 Python，不画耦合矩阵、不写三层次、不讨论涌现

### 例 5：用户说"给我们这台 TOF-MS 加一个实时峰识别 + 自动校准的 AI 层"

- **诊断**：OCGS（硬件 + 采集 + 标定 + 样品基质 + 操作员）
- **三层次**：
  - 哲学：识别哪些峰？为了什么科学问题（定量 / 鉴定 / 筛查）？
  - 基础：质谱物理（电离效率、空间电荷、漂移、分辨率）；噪声模型
  - 应用：网络架构、推理延迟、部署
- **总体设计**：耦合矩阵显式包括"标定漂移"、"基质效应"、"操作员行为"
- **综合集成**：领域专家给物理先验 → 小数据训练 + 物理仿真增广 → 残差按物理参数分组分析 → 改进先验
- **涌现警报**：识别错的峰会进入下次自动标定 → 反馈回路 → 必须有人工置信度门控

## 不要做的事

- **不要把这套方法变成"先讲一堆理论再写代码"**——理论是为了让代码更对，不是表演。能用一句话说完就别写五段。
- **不要在简单系统上滥用**——第一步诊断为简单就立刻退出本 Skill。**这条比启用本 Skill 更重要。**
- **不要省略 OCGS / 复杂系统上的总体设计**——技术债以复利累积。
- **不要把"涌现"当万能借口**——一定要具体到哪个反馈回路、哪个耦合、哪条不变式。
- **不要照搬例子里的清单**——例子是为了示范思维，不同系统的清单内容不同。
