# 耦合矩阵：填法、示例、陷阱

耦合矩阵是总体设计的核心产物。它是一个 N×N 表（N = 子系统数），每个格子标注两两之间的耦合**类型**与**强度**。

## 为什么要做耦合矩阵

软件架构图通常只画"调用关系"或"分层关系"，**会漏掉**：

- 时序耦合（先后顺序、节奏）
- 状态耦合（共享配置、缓存）
- 物理耦合（仪器领域：共享硬件、热、电、机械）
- 标定耦合（A 的标定影响 B 的解释）
- 隐性反馈（A 的输出经过外界进入 B 的输入）

这些"看不见"的耦合是涌现 bug 的主要来源。耦合矩阵强制把它们列出来。

## 耦合类型清单（按需选用）

| 类型 | 含义 | 软件例子 | 仪器例子 |
|---|---|---|---|
| **数据耦合** | 共享数据 schema / 数据流 | 服务 A 写表，服务 B 读 | 探测器输出 → 分析模块 |
| **控制耦合** | A 调用、触发、命令 B | RPC、消息队列 | 控制器命令电机 |
| **时序耦合** | A 必须在 B 之前/之后/同步 | 双写一致性 | 触发同步、采样时钟 |
| **物理耦合** | 共享物理资源 | 共享 GPU、磁盘 IO | 共享电源、散热、机械支架 |
| **电磁/热耦合** | （仪器专属） | — | 一个模块发热影响另一个模块的温漂 |
| **标定耦合** | A 的校准影响 B 的解释 | （罕见） | 探测器增益标定影响定量准确性 |
| **状态耦合** | 共享可变状态、配置、缓存 | 全局 feature flag | 仪器运行模式 |
| **反馈耦合** | A 的输出回到 A 的输入（经过 B） | 推荐系统看用户反应改推荐 | 控制环路 |
| **人因耦合** | 操作员行为是耦合的一部分 | UI 改变操作习惯 | 操作员手动校准 |

## 强度分级

简化为四级，主观估计即可（关键是显式标注，不是精确）：

- **0 / 无**：可以独立部署、独立失败、独立修改
- **1 / 弱**：通过稳定接口耦合，一方变动另一方可平稳跟进
- **2 / 中**：耦合多种类型，变动需要协调
- **3 / 强**：紧耦合，一方失败/变动会立即影响另一方；通常需要总体设计统一规划

## 标注规则

每个非对角格子写：`[类型缩写]:[强度]`，可叠加。例：

```
A → B:  D:2, T:3      （数据耦合中等 + 时序耦合强）
B → C:  S:1           （状态耦合弱）
```

矩阵通常**非对称**：A 影响 B 与 B 影响 A 强度不同。

## 完整示例：质谱仪 AI 峰识别

子系统：
- I (Ionization 离子源)
- M (Mass analyzer 质量分析器)
- D (Detector 检测器)
- A (Acquisition 数据采集)
- C (Calibration 标定)
- P (Peak picking AI 峰识别)
- U (User / 操作员)

```
       │  I  │  M  │  D  │  A  │  C  │  P  │  U  │
   ────┼─────┼─────┼─────┼─────┼─────┼─────┼─────┤
   I   │  ─  │ D:3 │ Ph:2│  ─  │ Cal:3│  ─  │ Ctrl:2│
   M   │     │  ─  │ D:3 │  ─  │ Cal:3│  ─  │ Ctrl:2│
   D   │     │     │  ─  │ D:3, T:3│ Cal:3 │  ─  │  ─  │
   A   │     │     │     │  ─  │ D:2  │ D:3 │  ─  │
   C   │ Cal:3│Cal:3│Cal:3│ Cal:3│  ─  │ Cal:3│ Ctrl:2│
   P   │     │     │     │     │     │  ─  │ Fb:2 │
   U   │     │     │     │     │     │     │  ─  │
```

（图例：D=Data, T=Time, Ph=Physical, Cal=Calibration, Ctrl=Control, S=State, Fb=Feedback）

**从矩阵读出的关键风险**：

1. **C（标定）行/列全是 3**：标定耦合渗透整个系统。这意味着 P（峰识别）**不能假设标定是冻结的**，必须显式处理标定漂移与重标定。
2. **D → A 是 D:3 + T:3**：检测器到采集是数据 + 时序双强耦合，接口必须把时序契约写死（采样率、触发同步、buffer 语义）。
3. **P → U 有 Fb:2**：峰识别结果会通过用户决策影响后续采样/标定 → 反馈回路。必须设计置信度门控与人工复核。

**没有耦合矩阵时**，P（峰识别）会被当作"读数据出结果"的纯函数，结果就是上线后标定一漂、基质一变就崩。

## 常见陷阱

1. **只画"模块依赖图"（控制耦合）**：漏掉其他类型。强制过一遍上面的类型表。
2. **认为"我们用消息队列了所以解耦了"**：消息队列只解耦了控制和时序的一部分，数据/标定/状态/反馈耦合都还在。
3. **强度全标 2**：典型的"不愿做判断"。强迫自己每行至少标出一个 3 或多个 0，才能逼出真正的耦合热点。
4. **矩阵写了不维护**：每次大改动都要更新矩阵；矩阵和代码不同步等于没有。
5. **忘记把"人/环境"作为子系统**：OCGS 的关键特征。物理仪器尤其要把操作员、样品、实验环境作为系统的一部分。

## 极简变体

如果懒得画完整矩阵，至少写出**耦合最强的 3 对子系统**及其耦合类型，并为每对写一段"接口契约"。这是最低底线。
