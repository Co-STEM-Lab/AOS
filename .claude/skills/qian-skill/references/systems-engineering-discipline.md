# 系统工程纪律：目的性 · 双想法 · 一次成功 · 系统层次

钱学森的系统工程实践（"两弹一星"、航天系统工程）沉淀出几条**纪律性原则**，移植到现代软件工程同样犀利。

## 1. 目的性 / 目标函数（Purposefulness）

**钱学森**：任何工程系统必须有明确的**目标**（target function），其他一切都是为目标服务的子目标。

**软件工程的常见失败**：

- 没有目标函数：在 latency、cost、reliability、developer experience 之间漫无目的地优化，每一项都"改善了一点"但整体没变好
- **隐式**目标函数：团队默认在优化某个指标（比如"代码行数"或"PR 数量"），但没人把它说出来，所以也没人质疑
- **错的**目标函数：优化 CI 通过率而非软件正确性 → Goodhart 法则发作

**纪律**：
- 每个工程项目必须显式写出 ≤3 个**系统级**目标，并标注它们之间的取舍
- 任何子优化都要先回答"它服务哪个系统目标"
- 警惕：当指标本身变成了目标，它就不再是好指标（Goodhart's law）

**实操**：
```
项目 X 的目标函数（按优先级排序）：
  1. 端到端 P99 延迟 < 500ms（硬约束）
  2. 故障恢复 MTTR < 5 分钟
  3. 开发体验：本地启动 < 60 秒
取舍：宁可慢一点也要可恢复（2 > 1 的优先级处理冲突）
```

## 2. 双想法 (Forward / Backward thinking)

**钱学森在航天工程总结**：每个设计要做"双向想"：

- **正想**：从目标到方案，一步步推下去："要达成 X，需要 Y，需要 Z"
- **反想**：从失败到原因，一步步反推："如果 X 失败了，可能是 Y 错了、Z 错了"

**对应软件工程**：

| 思维方向 | 软件工程对应 |
|---|---|
| 正向 | requirement → design → implementation → test |
| 反向 | "what could go wrong" → 防御性设计 → 故障注入 → DR drill |

钱学森的洞见：**只做正向是远远不够的**。航天系统不能"上去再修"，所以反向思考是强制纪律。

**软件工程版反向清单**（开发前过一遍）：

- 这个功能在什么输入下会爆？（边界、负值、空、NaN、超长、特殊字符）
- 在什么环境下会爆？（断网、慢盘、OOM、时钟回拨、依赖不可用）
- 在什么并发模式下会爆？（race、deadlock、starvation、thundering herd）
- 它失败时是**快速失败**还是**静默退化**？（哪种对系统更安全？）
- 它失败时**谁会被通知**？通知给的人**能修**吗？

## 3. 一次成功（First-time success mindset）

**钱学森**：航天系统的特点是"上去就不能 in-place 修"，所以工程文化必须围绕**一次成功**展开。

- 不依赖"上线后再说"
- 不依赖"线上 hotfix"
- 不依赖"如果坏了就回滚"

**软件工程的现实**：大部分场景没那么严苛（可以 rollback、可以 canary、可以 hotfix），但**当部署/操作代价高时**，应该切换到"一次成功"模式：

| 高代价场景 | 该用"一次成功"心态 |
|---|---|
| 数据库 schema migration | 是 |
| 修改生产数据 | 是 |
| 修改不可回滚的配置 | 是 |
| 跨团队 / 跨服务的协议变更 | 是 |
| 部署到边缘设备 / 嵌入式 | 是 |
| 修改用户面 API（已有用户） | 是 |
| 本地 dev 改代码 | 否（迭代代价低） |
| 内部服务的小 PR | 否 |

**一次成功的工程化**（钱学森"两弹一星"经验提炼到软件）：

1. **同行评审 ≥2 人**：不止 reviewer，还要有架构师 + 领域专家
2. **走查（walkthrough）**：把"如果我是用户/操作员，我会怎么做"完整模拟一遍
3. **失败模式分析**：写下来"会不会因为 X 失败"，针对每个 X 设计防护
4. **dry-run / staging 完整复现**：不接受"我本地跑过"
5. **失败时的状态可恢复性**：哪怕一次没成功，也要能优雅退出

**反模式**：在高代价场景下用"快速迭代"心态。在低代价场景下用"过度审查"。

## 4. 系统层次性（Micro / Meso / Macro）

钱学森在 OCGS 方法论中明确指出：**系统有层次，不同层次用不同方法**。

```
宏观（Macro）：整体行为、涌现性质、系统级 KPI
   ↑↓
介观（Meso）：模块、子系统、集群、团队
   ↑↓
微观（Micro）：单一组件、单一进程、单一函数
```

**软件工程映射**：

| 层次 | 对应 |
|---|---|
| 宏观 | 端到端体验、SLO、业务 KPI、组织文化 |
| 介观 | 服务、模块、subnet、团队 |
| 微观 | 函数、类、单一 RPC、单一 SQL |

**层次混淆是大坑**：
- 在微观层面用"业务 KPI"做指引 → 过度设计（一个函数加 10 层抽象因为"将来可能"）
- 在宏观层面用"代码行数"做指引 → 优化无关紧要的细节
- 在介观层面用"算法复杂度"做指引 → 漏掉跨服务的真正瓶颈

**纪律**：当你在分析或设计时，**明确说出你在哪一层**，并用对应的工具：

| 层次 | 工具 / 视角 |
|---|---|
| 微观 | profiler、debugger、type system、unit test |
| 介观 | distributed tracing、integration test、coupling matrix |
| 宏观 | KPI dashboard、SLO、chaos engineering、incident review |

**跨层次现象**（涌现）：宏观问题往往**没有**微观/介观原因。例如"用户增长停滞"在代码层面找不到原因——它在产品-用户-市场的宏观耦合里。**不要在微观找宏观问题的根因**。

## 5. "总体设计部"思想（General Designer Department）

钱学森在航天系统提出，复杂系统必须有专门的"总体设计部"：

- **不写代码**，专门做总体设计
- **掌握系统级目标**，协调子系统
- **拥有 veto 权**：可以否决违反总体设计的子系统决策
- **持续维护总体设计文档**，与实际系统保持同步

**软件工程版本**：

| 钱学森 | 软件 |
|---|---|
| 总师 | Tech lead / 架构师 / staff engineer |
| 总体设计部 | Architecture review committee / tech council |
| 总体设计文档 | RFC / ADR / system design doc |
| Veto 权 | 架构审查的"block" 权限 |

**现代软件工程的常见失败**：
- 没有总师 / 总师没有 veto 权 → 每个团队各自最优，系统级失败
- 总体设计文档不维护 → 文档和实际系统漂移，新人无法理解
- 总师写太多代码 → 失去"超脱视角"

**钱学森经验**：总师**必须懂行**，但**必须不直接写**关键组件，否则丧失全局视角。

## 整合到本 Skill 五步流程

这份文档的几条原则散布在 SKILL.md 的不同步骤里：

- **目的性 / 目标函数** → 第三步总体设计清单的第 5 项（系统级指标）
- **双想法** → 第四步综合集成的"列假设"机制
- **一次成功** → 第五步涌现检查（每次大改动后）
- **系统层次** → 第一步诊断 + 第二步三层次（哲学/基础/应用）
- **总体设计部** → 第三步整体精神

但单独成文有价值，因为这些是**横切的工程纪律**，应该被工程师内化，而不是每次查 SKILL.md。
