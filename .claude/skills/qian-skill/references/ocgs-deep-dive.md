# 开放复杂巨系统（OCGS）深入

## 钱学森的原始定义

钱学森、于景元、戴汝为在 1990 年《一个科学新领域——开放的复杂巨系统及其方法论》中提出了 OCGS 的判据：

1. **巨**：子系统数量极大（不是十几个，是成千上万乃至更多）
2. **复杂**：子系统**种类多**且**层次多**，相互关系也复杂（不是同质大量）
3. **开放**：系统与外界（包括人）有物质、能量、信息交换
4. **行为不可纯还原**：整体行为无法通过研究各子系统行为简单相加得到

这四条同时满足，称为 OCGS。

经典例子（钱学森给出）：人体、生态系统、社会经济系统、宇宙、人脑。

## 迁移到软件 / 仪器领域的判据松弛

严格按钱学森的原始定义，软件系统少有真正的 OCGS（子系统数量未必"巨"）。但**他的方法论价值不在严格分类，而在工程态度的转变**：

- 一旦系统具有**多种异质子系统 + 强耦合 + 与人/物理环境的开放性**，就应该**采用 OCGS 方法论**，不必纠结子系统是否上万。
- 实操中我们把这种"具备 OCGS 关键特征"的系统也称为 OCGS。这是工程化的简化。

## 软件 / 仪器领域 OCGS 判据（工程化）

四条**同时满足**，按 OCGS 处理：

1. **异质性**：子系统至少 3 种不同性质（如：硬件、固件、控制环、ML 模型、数据库、UI、用户）
2. **强耦合**：至少一对子系统之间存在非平凡耦合（不只是函数调用，还有时序、状态、物理、标定等）
3. **开放性**：系统与外界有持续交换，**且其中至少包含人或物理环境**
4. **不可纯还原**：你**写不出**一个能从子系统行为推出系统行为的解析模型

第 4 条是关键的"软"判据，可以用反证：如果你能为系统写出端到端的可解析模型，那它就不是 OCGS。

## 为什么这个分类重要

不同类型适用不同方法，**错配代价高昂**：

- 把简单系统当 OCGS：过度设计、过度文档、速度拖慢，方法论开销 > 价值
- 把 OCGS 当简单系统：**这是大多数复杂软件项目失败的根因**——
  - 各子系统单元测试都通过，集成就崩
  - 性能瓶颈在跨子系统耦合处，单子系统优化无效
  - 修一个 bug 引入三个新 bug（涌现）
  - "重写一遍就好"——但重写后同样会失败，因为方法没变

## 与"复杂系统"科学的关系

OCGS 与 Santa Fe Institute 学派的"复杂系统"（complex systems）/ 复杂适应系统（CAS）有重叠也有不同：

- 重叠：都重视涌现、非线性、自适应、不可还原
- 不同：钱学森更**工程化**，强调"如何对 OCGS 做工程"（综合集成法、总体设计部），而 CAS 学派更偏**理论与建模**

对 AI 编码场景，钱学森派的态度更可操作：**承认复杂，然后用人 + 机 + 数据三方协同把工程做下去**，而不是寄希望于找到一个干净的理论模型。

## 反例：什么不是 OCGS

- **一个 CRUD app**：子系统同质（CRUD + DB），无开放性中的"人/物理耦合"，可以纯还原。**不是 OCGS**。
- **一个编译器**：子系统多但封闭（lex / parse / typecheck / IR / codegen），无人/环境持续耦合，可解析模型存在（语言语义）。**不是 OCGS**。但**编译器+IDE+用户**整体可能是 OCGS。
- **一个深度学习模型**：单一子系统，不是系统。**不是 OCGS**。但**模型 + 训练集 + 反馈数据回流 + 用户**可能是。

注意：判定本身有边界含糊性，重要的是**意识到你在做的工程是哪一类**，而不是死抠分类。
