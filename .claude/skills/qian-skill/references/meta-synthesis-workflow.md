# 综合集成法（Meta-Synthesis）工作流详解

钱学森的综合集成法（从定性到定量综合集成法）是 OCGS 的核心方法论。本文档把它落到 AI 协同编码场景。

## 核心思想

对 OCGS，**纯定量**（数据驱动、纯算法）和**纯定性**（专家拍脑袋、理论推演）都不够，必须把三者综合：

```
专家经验 / 领域理论（定性）
            +
真实系统数据 / 仿真（定量）
            +
研讨循环 / AI 协同（综合）
            =
带置信度的工程决策
```

钱学森把这个循环称为 **"从定性到定量综合集成研讨厅"** ——一个把人、机器、数据放在同一个工作流的"厅"。

## 在 AI 编码场景里，"研讨厅"是谁

| 角色 | 在传统研讨厅中 | 在 AI 编码中 |
|---|---|---|
| **领域专家** | 物理学家、工程师 | 工程师、领域专家、用户 |
| **数据/仿真** | 数据库、仿真系统 | 日志、profile、trace、回放数据、单元测试、CI |
| **协调者** | 总体设计部 | **你（AI）作为协同者** |
| **决策者** | 总师 | 工程师 |

注意：**AI 不是研讨厅的"专家"**，而是**协调者 + 数据加工者**。你不能替代领域判断，但能加速假设验证、信息整合、跨系统搜索。

## 三个完整 walkthrough

### Walkthrough A：跨子系统 bug 诊断

**情境**：用户报告"分布式训练 job 跑着跑着 hang，但单机能跑"。

**错误做法**（纯定量 / 纯 AI）：直接 grep 报错、加 try-except、加重试、加 timeout。

**错误做法**（纯定性）：工程师拍脑袋说"网络问题"，加大 timeout。

**综合集成做法**：

1. **定性枚举假设**（人 + AI 协同列出 3-5 个，不要少于 3）：
   - H1: 网络抖动 → NCCL 集合通信卡住
   - H2: 数据加载器进程僵死 → GPU 饿死
   - H3: 通信原语死锁（不同 rank 调用顺序不一致）
   - H4: OOM 触发 OOM killer → 个别进程消失但其他在等
   - H5: 文件系统/存储瓶颈

2. **为每个假设设计最便宜的诊断**（定量验证步骤）：
   - H1 → 跑 NCCL test、检查 nccl-debug log
   - H2 → py-spy dump 所有 worker、看是否在 `_get_batch`
   - H3 → 检查 all_reduce 顺序、看有无条件分支导致 rank divergence
   - H4 → dmesg、nvidia-smi、各 rank 的 ps aux
   - H5 → iostat、网络存储延迟

3. **按"代价低 + 排除力强"排序执行**：先看 dmesg（10 秒）、再看 py-spy（1 分钟），再跑 NCCL test（5 分钟）。**避免过早跳到代价高的诊断**。

4. **每一步**：要么排除一个假设、要么收敛到一个。把结果**写下来**（trace.md 或 PR 描述），让协作者能复现你的推理。

5. **找到根因后**：不仅修代码，**回到第三步的总体设计（耦合矩阵）**，把这条新发现的耦合补进去。

**关键纪律**：
- 不要在没有列假设前就改代码
- 不要只验证一个假设就收手
- 不要让"看起来更可能"的假设挤掉其他验证（确认偏差）

### Walkthrough B：科学仪器 AI 模块设计

**情境**：给质谱仪写一个 GC-MS 峰识别神经网络。

**错误做法**：直接拿现成 1D-CNN 跑数据集、调参刷指标。

**综合集成做法**：

1. **定性先验（领域理论层）**——先和领域专家把以下写清楚：
   - 物理：峰的形状（高斯/EMG）、峰宽与质量分辨率的关系、同位素模式、共流出
   - 噪声模型：散粒噪声、电子噪声、化学背景
   - 标定漂移：保留时间漂移、质量漂移的典型尺度
   - 样品基质效应：基质如何影响峰形与定量
   - **应用约束**：定量精度要求？检测限要求？实时性？

2. **基线（应用层）**——传统算法的性能基线，必须做：
   - 经典峰识别（如 ApexTrack、CWT）的性能、失败模式
   - 这是你的下界。AI 模型必须显著优于它，否则不值得做。

3. **数据策略（定量层）**：
   - 真实数据 + **物理仿真增广**（用第 1 步的物理模型生成可控合成数据）
   - 评测集必须分层（按基质、浓度、保留时间区间）
   - 漂移测试集（不同日期、不同仪器）

4. **模型设计**：把第 1 步的物理先验**注入模型**——例如不变性（保留时间平移不变性）、单调性约束（强度不能为负）、多任务（同时输出峰位置与峰形参数，可与物理校核）。

5. **训练与评测循环**（综合集成的主循环）：
   - 训练 → 在分层评测集上跑 → **残差按物理参数分组**
   - 残差分析回到第 1 步：是哪种物理情况误差大？为什么？
   - 调整数据 / 先验 / 架构 → 再训

6. **部署与反馈**：
   - **置信度门控**：低置信度的峰进入人工复核队列
   - **标定漂移监测**：定期对照标准品，发现漂移触发再校准
   - **闭环风险**：人工复核的结果可能进入下一轮训练数据 → 显式管理这个反馈回路（不要让坏数据进训练集）

**关键纪律**：
- 不要让"baseline 性能" 是一个 SOTA 神经网络（你需要的是经典算法的下界）
- 不要在没有残差分析的情况下"再炼一版"
- 不要忽视部署后的反馈回路

### Walkthrough C：大型软件系统重构

**情境**：把一个 monorepo 从自研构建脚本迁移到 Bazel。

**综合集成做法**：

1. **定性目标（哲学层）**：
   - 为什么重构？CI 时间？开发体验？缓存命中率？跨语言一致性？
   - 哪些**不**重构（不变量）：用户面 API、版本号语义、发布流程

2. **现状定量（数据层）**：
   - 当前 CI 时间分布（按步骤、按 PR 类型）
   - 当前缓存命中率
   - 当前最痛的开发者投诉（搜 Slack / issues）

3. **耦合矩阵**（总体设计层）：
   - 包之间的依赖图（这是显式的）
   - 包与 CI 步骤的耦合（隐式：哪些包改动会触发整个 CI 重跑）
   - 包与人/团队的耦合（社会耦合）

4. **方案对比**（综合定性 + 定量）：
   - 列 ≤3 个备选（Bazel / Buck / 自研改良）
   - 对每个写一段定性论证（开发体验、生态、成本）
   - 对每个写一段定量预测（CI 时间改进、缓存命中率改进，**用现状数据外推**）

5. **小范围试点**：选一个**耦合弱**的子图先迁，量化结果，比对预测。预测偏差大就回到第 4 步。

6. **涌现监测**：迁移过程中观察的非预期：
   - 开发者本地构建习惯有没有改变？
   - 出现了什么新的 CI 不稳定模式？
   - 团队之间的协作摩擦如何变化？

**关键纪律**：
- 不要在没有现状定量基线时就开始迁移（无法判断进步）
- 不要全量迁移（涌现风险太高）
- 不要忽视社会耦合（人是 OCGS 的一部分）

## 综合集成的反模式（常见错误）

1. **跳过定性枚举**：直接搜代码 + 改 → 大概率改错地方
2. **跳过定量验证**：领域专家说"应该是 X"就改 → 经常是 X' 而非 X
3. **过早收敛**：第一个看起来对的假设就停 → 漏掉真因
4. **缺少残差/回顾循环**：一次改完不复盘 → 同类问题再发生
5. **AI 单跑**：脱离人类领域直觉，把"代码看起来对"当成"对" → 系统级失败
6. **人单跑**：忽视数据 → 改的是想象中的系统

## 一句话总结

**让人提假设、让数据做裁判、让 AI 做协调。** 三者缺一不可，且循环进行，不是一次性瀑布。
